01 / Проблема

Бюджет не сходился с обещанным объёмом работ

Крупный заказ судостроительного завода на линию плазменной резки профиля. Бюджет заказчика не сходился с обещанным объёмом — искали, где снизить стоимость. Блок «программа и диспетчеризация» сторонний подрядчик оценивал в 3 миллиона рублей. Я взял разработку на себя.

Задача внутри блока: у инженера-технолога есть 2D-чертежи деталей — их надо превратить в проверенное задание для линии.

  • Шесть типов судового профиля: полособульб, гнутый швеллер, равнополочный и неравнополочный уголок, полоса, тавр. У каждого своя геометрия сечения.
  • Десятки типов резов и маркировок, у каждого свои параметры — углы, длины, привязки к граням.
  • Деталь с пересекающимися резами не должна попасть в производство: испорченный хлыст профиля — это деньги и сорванный срок.
  • Детали надо разложить по хлыстам так, чтобы отход был минимальным.
02 / Ключевая идея
Каждый рез оставляет после себя тело. Проверка пересечений — это расстояние между телами.

Профиль строится как твердотельная модель, рез применяется к ней булевой операцией — это очевидно. Неочевидно другое: перед резом мы запоминаем тело, после реза вычитаем одно из другого и получаем солид удалённого металла — физическую форму того, что вырезал резак.

Дальше всё становится простым. Вопрос «не пересекаются ли два реза» превращается в расстояние между двумя телами. Не формула для полособульба, отдельная формула для гнутого швеллера и ещё одна для уголка — одна проверка на все шесть типов профиля и на все типы резов сразу. Визуализация достаётся почти даром: модель уже построена, её остаётся отдать в браузер.

backend/core/utils/math_helpers.py — извлечение удалённого металла
solid_before = wp.val()
wp_after, cut_meta = cut_fn(wp, **kwargs)

removed = solid_before.cut(wp_after.val())   # тело удалённого металла
if removed.Volume() > 1e-6:
    return wp_after, cut_meta, removed
backend/core/validator.py — вся проверка пересечения резов
for existing_id, existing_solid in self._cut_solids.items():
    dist = removed_solid.distance(existing_solid)
    if dist < COLLISION_DISTANCE_MM:          # 2 мм
        return CollisionResult(
            collision_type="cut_cut",
            conflict_cut_id=existing_id,
            distance=dist,
        )

self._cut_solids[cut_id] = removed_solid

Фрагменты сокращены: убраны обработка исключений OpenCASCADE и тексты сообщений. Проверка работает для любого профиля и любой пары резов, потому что оперирует не чертежом, а металлом.

03 / Архитектура

От параметров детали до карты раскроя

2D-чертёж → параметры детали профиль · длина · сечение · резы · маркировка  /  React + Zustand
FastAPI → CadQuery в пуле потоков геометрия синхронна и CPU-bound: в event loop она бы всё остановила
твердотельная модель профиля → резы булевыми операциями каждый рез отдаёт тело удалённого металла
GLB → 3D в браузере React Three Fiber, подсветка конфликтов
проверка коллизий расстояние между телами < 2 мм
деталь → PostgreSQL проекты, чертежи, детали, карты раскроя
раскрой → OR-Tools CP-SAT ширина реза + форма торцов деталей
карта раскроя → CSV согласованного формата → на линию

Модель — единственный источник истины: и для картинки, и для проверок.

04 / Раскрой

Задача об упаковке, у которой детали с косыми торцами

Разложить детали по хлыстам профиля — на вид школьная задача об упаковке. На деле у деталей косые торцы: два соседа со встречными скосами придвигаются друг к другу вплотную, а два одинаковых — нет. Зазор между деталями зависит от того, какими торцами они повёрнуты друг к другу, плюс ширина самого реза.

Поэтому это не жадная эвристика, а задача комбинаторной оптимизации: OR-Tools CP-SAT в формулировке через маршруты — хлыст это «машина», детали на нём — «маршрут». Симметричные детали разрешено переворачивать, асимметричные — нет. Солвер возвращает OPTIMAL, если доказал оптимальность, и FEASIBLE, если упёрся в лимит времени.

05 / Что система делает

Функции, доведённые до пользователя

  • строит параметрическую твердотельную модель шести типов судового профиля;
  • применяет резы и маркировку как булевы операции, извлекая тело удалённого металла;
  • проверяет пересечения резов, наложение маркировок и корректность параметров;
  • показывает деталь в браузере в 3D с подсветкой конфликтов;
  • хранит проекты, чертежи, детали и карты раскроя в PostgreSQL;
  • раскладывает детали по хлыстам солвером CP-SAT с учётом ширины реза и формы торцов;
  • выгружает карту раскроя в CSV согласованного с линией формата и печатает PDF.
06 / Границы решения

Чего система не делает — и почему это осознанно

Инженерное решение честнее описывать вместе с его границами. Вот что осталось за рамкой блока.

не читает чертежи

Параметры переносит человек. Распознавание DWG/DXF — отдельный проект, в бюджет блока он не входил.

не генерирует G-код

Выход — CSV согласованного с линией формата. Стойка ЧПУ остаётся за границей системы.

не моделирует резак

Проверяются коллизии резов между собой, а не столкновения резака с оснасткой. Каркас для запретных зон профиля заложен, зоны ещё не заданы.

не проверена на металле

Пайплайн доведён до CSV. Приёмка на линии впереди.

07 / Метод разработки

Блок под команду, закрытый одним человеком

Работа заняла 26 активных рабочих дней — 210 коммитов с апреля по июль 2026. 218 файлов, 31 494 строки, из них 7 901 — тесты.

Метод — разработка с AI-агентами. Что сделало её управляемой на проекте такого размера: требования с идентификаторами прямо в шапках модулей, Pydantic-схемы и словарь типов резов как единственный источник истины, тесты на бэкенде и фронте, миграции Alembic, жёсткая нарезка на модули — профиль, рез, солвер, экспорт.

Агент пишет быстро и уверенно — в том числе там, где ошибается. Поэтому последний контур контроля остаётся человеческим: архитектурные решения и чтение диффа я не делегирую.
