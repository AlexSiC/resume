## Кейс 1. Видеобаза знаний инженерной компании

**Статус:** черновик текста ✅ · схема ⬜ · скринкаст ⬜ · цифры ⬜

### Проблема
В инженерной компании много видео: зум-встречи, апробации линий, испытания оборудования. Знания в них реально ценные — и полностью недоступные: чтобы найти «где обсуждали тот узел», надо пересматривать часы записей. Уходит сотрудник — уносит контекст. Запрос пришёл от технического директора.

### Решение
Построил всю архитектуру и инфраструктуру системы:
Сотрудник снимает видео и кидает его в **Telegram-бот** → видео попадает в **управляемую базу данных с хостингом** → каждое видео/аудио **автоматически транскрибируется** → транскрипт ложится в **RAG-базу** → инженер задаёт вопрос обычным языком и получает ответ **со ссылкой на видео с релевантным фрагментом**.

Порог входа для сотрудника — ноль: отправить видео в телеграм умеют все. Никаких форм, тегов и обязательных регламентов.

`[схема: Telegram-бот → хостинг + БД → транскрибация → RAG → поиск/ответ со ссылкой]`

### Результат
- Видео перестало быть «мёртвым архивом»: по нему можно искать и задавать вопросы.
- Знание остаётся в компании даже при уходе сотрудника.
- `[уточнить: сколько видео/часов в базе, пример реального запроса, время поиска до/после]`

### Стек
Python, Telegram Bot API, транскрибация `[уточнить: Whisper?]`, векторная БД, LLM, видеохостинг.

## Кейс 2. RAG-база знаний со смысловым чанкованием

**Статус:** черновик текста ✅ · схема ⬜ · скринкаст ⬜ · цифры ⬜

### Проблема
Компания хочет «чёрный ящик»: задать вопрос о любом опыте компании и получить достоверный ответ — не изобретать велосипед и не отрывать дорогостоящих техлидов. Но документы компании неструктурированы, а классический RAG с механической нарезкой на чанки рвёт смысл: одна тема размазана по разным местам документа.

### Решение — с честной историей неудач
Прошёл три подхода:
1. **Написал GraphRAG самостоятельно** — техническая связка несложная, но качественный граф не построился: узкое место — извлечение сущностей и связей.
2. **Взял GraphRAG от Microsoft** (open source) — та же проблема: онтологию сущностей и связей для этого домена непонятно как задать.
3. **Вернулся к классическому RAG на умных моделях** — и решил настоящую проблему: чанкование. Создал pipeline: на входе любой (даже плохо структурированный) документ → на выходе **JSON с чанками, собранными синтетически по смыслу** — если одна тема обсуждается в разных местах документа, дословные фрагменты сводятся в один чанк.

Вывод, который экономит заказчику деньги: в RAG вся суть не в векторной базе, а в том, как готовятся данные. Знаю пределы применимости GraphRAG — и когда он не нужен.

`[схема: документ → LLM-пайплайн смыслового чанкования → JSON чанков → векторная БД → ответы]`

### Результат
- По обработанным документам RAG выдаёт точные ответы `[уточнить: пример вопрос-ответ]`.
- Побочный продукт: RAG-база по инженерным учебникам (сопромат, теормех, физика) — конструкторы быстрее находят справочную информацию `[уточнить: насколько]`.

### Стек
Python, LLM (Claude, GPT), эмбеддинги, векторные БД, JSON-пайплайны.

### Доделать
- [ ] Схема пайплайна чанкования
- [ ] Скринкаст: документ на входе → JSON чанков → вопрос → точный ответ (на синтетическом/открытом документе)
- [ ] Пример «вопрос → ответ» для текста кейса

## Кейс 3. ПО управления линией плазменной резки профиля

**Статус:** черновик текста ✅ · схема ⬜ · скринкаст ⬜ · цифры ⬜

### Проблема
Крупный заказ: бюджет заказчика не сходился с обещанным объёмом. Искали, где снизить стоимость, — блок «программа и диспетчеризация» подошёл. Взял разработку на себя. Задача: инженер-технолог заказчика имеет 2D-чертежи деталей — их надо превратить в параметры для станка плазменной резки профиля.

### Решение
Разработал ПО промышленного уровня, ключевое инженерное решение — **твердотельное моделирование резов на CadQuery**: движку описываешь рез командами, он строит твердотельную модель — и проверка пересечений резов и выхода за границы получается лаконичной и надёжной, а визуализация — почти бесплатной, потому что модель уже есть. Плюс UI, база данных и **задача раскроя**: как разложить максимум деталей на одном хлысте профиля.

Метод разработки — вайбкодинг с AI-агентами: один человек закрывает блок, под который закладывалась команда.

`[схема: 2D-чертёж → параметризация резов → CadQuery-модель → проверки → раскрой → параметры для станка]`

### Результат
- Стоимость проекта для заказчика снижена — блок закрыт одним человеком `[уточнить: на сколько %/₽]`.
- Готовность ~80% `[обновить по факту]`.

### Стек
Python, CadQuery (твердотельное моделирование), UI `[уточнить фреймворк]`, БД, алгоритмы раскроя, AI-агенты как метод разработки.

## Кейс 4. Техническое зрение для растарки мешков

**Статус:** черновик текста ✅ · схема ⬜ · скринкаст ⬜ · цифры ⬜

### Проблема
На производственную линию поступают мешки 20–50 кг. Ручная растарка — тяжёлый и травмоопасный труд; чтобы снимал робот, ему нужны точные координаты каждого мешка, а мешки лежат неровно, слипаются, мнутся — классическая задача, где «жёсткая» автоматика не справляется.

### Решение
Сам разработал программно-аппаратный комплекс: камера над линией → **обученная нейросеть** определяет положение и координаты мешков → координаты передаются роботу для съёма. Прошёл весь цикл: сбор и разметка данных, обучение модели, интеграция с железом.

`[схема: камера → нейросеть (детекция) → координаты → робот]`

### Результат
- Получен **работающий proof of concept** программно-аппаратного комплекса.

### Стек
